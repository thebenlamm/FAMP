<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| installer → launchd/systemd (first load) | This plan actually LOADS the auto-launch service. The load command runs only after guardian sign-off (Plan 03). |
| install-time environment → bind() | A sandboxed shell cannot bind the socket; installing there would write a service that can never work. |
| installer → loginctl (Linux linger) | linger is a per-user system policy; auto-escalating it is forbidden. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-09 | Tampering | First service load before review | mitigate | depends_on Plan 03 (guardian sign-off) — wave ordering guarantees no `launchctl bootstrap` precedes the approved plist. |
| T-05-10 | Denial of Service | Installing in a sandbox writes a service that can never bind (silent broken state) | mitigate | BOOT-02: check_not_sandboxed reuses Phase 4 EPERM-on-bind probe; install refuses with non-zero exit + guidance before writing anything. |
| T-05-11 | Elevation of Privilege | Silently running `loginctl enable-linger` changes per-user system policy; banned on hardened hosts | mitigate | D-08: detect-and-instruct ONLY — install PRINTS the `enable-linger` command, never runs it. Acceptance asserts no loginctl-enable-linger Command invocation exists. |
| T-05-03 | Spoofing | Two-brokers-bind-same-socket steal race | accept | OUT OF SCOPE (deferred spawn-lock track, per ROADMAP). Known accepted risk; daemon's connect-first short-circuit dissolves it for daemon users. Not mitigated here. |
</threat_model>
