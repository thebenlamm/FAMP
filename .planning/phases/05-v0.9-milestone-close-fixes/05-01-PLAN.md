<objective>
Repair the CC-07 latent bug surfaced by `.planning/v0.9-MILESTONE-AUDIT.md`:
the `/famp-who [#channel?]` slash command currently declares
`mcp__famp__famp_sessions` in `allowed-tools` and instructs the model to call
that tool when the argument is `#<channel>`. The MCP server registers only 8
tools (`famp_send`, `famp_await`, `famp_inbox`, `famp_peers`, `famp_register`,
`famp_whoami`, `famp_join`, `famp_leave`); `famp_sessions` is not among them, so
`/famp-who #foo` from Claude Code returns an unknown-tool error.

Purpose: Close milestone gap #1 (BROKEN) without expanding the MCP surface. The
fix is purely client-side in the slash-command markdown — call `famp_peers` for
both branches and project the channel filter in the model instructions.

Output: A repaired `famp-who.md` asset shipping inside the `famp` binary and a
regression test that locks the constraint in place.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → Claude Code → MCP slash-command markdown | The slash-command markdown is a model-instruction template, not executable code. The only mutable surface is the `$ARGUMENTS` value (typically `#<channel>` or empty). |
| Slash command → MCP tool dispatch | `allowed-tools` frontmatter is enforced by Claude Code's MCP layer; an unregistered tool name produces a typed error (the very bug this plan fixes). |
