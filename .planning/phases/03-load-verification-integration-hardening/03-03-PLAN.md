# Phase 03 Plan 03: Saturated Direct Inspect RPC No-Starvation Gap Closure

## Objective

Close `GAP-03-01` by making inspect snapshot work non-blocking for the broker's main message-processing path, then strengthening `inspect_load_test` so it proves the original Phase 3 success criterion under saturated direct inspect RPC pressure.

The current committed load test proves paced concurrent inspect pressure. A stricter direct-RPC variant observed `baseline=1582 loaded=262 ratio=0.17`, below the required `0.80`. This plan keeps the stronger product claim: inspect calls at saturating rate must not starve live bus traffic.

## Gap Being Closed

- `GAP-03-01`: Saturated direct inspect RPC load is not proven.
- Evidence source: `.planning/phases/03-load-verification-integration-hardening/03-VERIFICATION.md`
- Closure target: `cargo nextest run -p famp --test inspect_load_test --no-fail-fast` passes after the test is converted to direct saturated RPC calls.

## Design

`Out::InspectRequest` currently performs the 500ms `spawn_blocking` inspect dispatch and awaits it while `execute_outs` is processing broker outputs. Under saturated direct inspect RPC load, that await path can keep the broker from promptly processing ordinary send/receive traffic.

Change the broker so inspect requests capture an immutable snapshot, clone the requesting client's reply sender, acquire an inspect concurrency permit, and dispatch the blocking snapshot work from a spawned async task. The outer `execute_outs` loop should return to processing other broker outputs immediately.

Bound the new asynchronous inspect work with a small semaphore so direct inspect floods cannot create an unbounded queue of blocking tasks. Preserve the existing 500ms response budget and `BudgetExceeded` reply behavior.

## Threat Model

- **Threat: inspect RPC denial of service through unbounded spawned work.**
  - Mitigation: add a broker-local inspect concurrency limit and use non-waiting permit acquisition. If no permit is available, reply with the existing budget-exceeded payload instead of queueing more work.
- **Threat: stale or inconsistent inspect reads.**
  - Mitigation: keep the existing snapshot-based discipline. Capture `BrokerStateView`, cursor offsets, bus path, socket path, and inspect kind before spawning work.
- **Threat: reply misrouting after asynchronous dispatch.**
  - Mitigation: clone only the specific request client's `mpsc::Sender<BusReply>` before spawning. Do not access the broker's reply sender map from the spawned task.
- **Threat: regressing cancellation/task/message inspect semantics.**
  - Mitigation: rerun the focused inspect integration tests after the dispatch change.

## Tasks

### Task 1: Make inspect dispatch non-blocking and bounded

**Files:**

- `crates/famp/src/cli/broker/mod.rs`

**Read first:**

- `crates/famp/src/cli/broker/mod.rs`
- `crates/famp-inspect-client/src/lib.rs`
- `crates/famp-inspect-server/src/lib.rs`
- `crates/famp/tests/inspect_cancel_1000.rs`
- `.planning/phases/03-load-verification-integration-hardening/03-VERIFICATION.md`
- `.planning/phases/03-load-verification-integration-hardening/03-REVIEW.md`

**Implementation steps:**

1. Add a broker-local inspect concurrency limit, for example `MAX_CONCURRENT_INSPECT_REQUESTS`, sized above the current load test inspector count and low enough to protect the blocking pool from unbounded inspect floods.
2. Create a `tokio::sync::Semaphore` for inspect requests in the broker listener/run loop and thread it into the helper that processes `Out` values.
3. In the `Out::InspectRequest { client, kind }` arm, clone the target client's reply sender before spawning any asynchronous work.
4. Capture the current `BrokerStateView`, cursor offsets, bus directory, socket path, and owned inspect kind exactly as the existing blocking path does.
5. Try to acquire an owned inspect permit. If no permit is available, send `BusReply::InspectOk { payload: inspect_budget_exceeded_payload(...) }` to the cloned client sender and return without queuing work.
6. If a permit is available, `tokio::spawn` an async task that owns the permit, runs the existing `spawn_blocking` inspect context/dispatch path under the existing 500ms timeout, maps timeout/join/error results to the current payload behavior, then sends the reply to the cloned client sender.
7. Do not await inspect snapshot construction or inspect dispatch inside the outer `execute_outs` loop.
8. Preserve the existing inspect error logging and the 500ms budget semantics.

**Acceptance criteria:**

- `Out::InspectRequest` no longer awaits the inspect `spawn_blocking` result on the broker's main out-processing path.
- The implementation contains bounded inspect concurrency, not an unbounded `tokio::spawn` per request.
- Existing budget exceeded replies are still delivered as `BusReply::InspectOk`.
- `cargo nextest run -p famp --test inspect_tasks --no-fail-fast` passes.
- `cargo nextest run -p famp --test inspect_messages --no-fail-fast` passes.
- `cargo nextest run -p famp --test inspect_cancel_1000 --no-fail-fast` passes.

4. Remove the inspector loop's 1500ms sleep and avoid adding a replacement sleep/backoff; this test must apply saturated direct RPC pressure.
5. Keep the sender traffic and receiver accounting from the current test so the throughput ratio remains comparable.
6. Update comments and assertion text so they describe saturated direct inspect RPC pressure.
7. Keep `STARVATION_THRESHOLD` at `0.80`.

**Acceptance criteria:**

- `inspect_load_test.rs` contains direct `famp_inspect_client::connect_and_call` usage for inspector load.

- `cargo nextest run -p famp --test inspect_load_test --no-fail-fast` passes.
