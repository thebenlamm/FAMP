<interfaces>
<!-- Verified-against-binary facts. Use these verbatim; do NOT re-derive or paraphrase the command surface. -->

`famp -V` → `famp 0.11.0`

`famp daemon --help` lists exactly four subcommands: install, uninstall, status, restart.
  - install: "Install the FAMP broker as a persistent user-level service (launchd on macOS, systemd --user on Linux). ... Idempotent: re-running is safe when the service is already installed"
  - status: "three states: not-installed, installed-but-down, running ... Exits 0 when running, 1 when not installed, 2 when installed but the broker process is not responding"
  - restart: "Restart the daemon, picking up a new on-disk binary after `cargo install`"
  - uninstall: "Stops and removes the service file. Idempotent: safe to run even when the service is not currently loaded"

`famp broker --help`:
  - `--no-idle-exit`  Run the broker without the 300s idle self-terminate
  - broker is "Auto-spawned by `bus_client::spawn::spawn_broker_if_absent`; rarely invoked directly by humans"

Orthogonality (D-01): `famp daemon install` (broker PRESENCE via launchd/systemd) and
`famp install-claude-code` / `famp install-codex` (MCP WIRING) are SEPARATE subcommands,
NOT substitutes. The fresh-clone walkthrough requires BOTH a daemon install AND per-tool
MCP wiring. Do NOT collapse them into a single line.

Daemon persistence mechanism (from the locked plist shape, DAEMON-02): the macOS LaunchAgent
has `RunAtLoad=true` + `KeepAlive=true` → the broker survives reboot/logout. The bridge
(`famp broker --no-idle-exit`) is one unsandboxed foreground terminal process → it dies on
terminal-close / logout.

Sandbox boundary (BOOT-01/BOOT-02): Codex runs sandboxed; auto-spawn-on-register hits EPERM
on `bind()`. The daemon (run from a normal shell) is the broker the sandboxed Codex client
then connects to. The installer itself refuses to run inside a sandbox.
</interfaces>
