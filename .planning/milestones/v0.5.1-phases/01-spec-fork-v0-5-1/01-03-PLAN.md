    Replace the `## §6.1 Agent Card (revised)` stub with a normative rewrite of the Agent Card section:

    1. Remove any reference to a self-signature `signature` field on the Agent Card (v0.5 has this; v0.5.1 retracts it).
    2. Add two required fields (per D-13 and RESEARCH §8 gap 2 naming decision — accept `federation_signature` as the signature field name):
       - `federation_credential` (required, string) — an opaque identifier referencing the federation-scoped credential used to sign the card. Format is federation-specific (opaque to FAMP); the only normative constraint is that the identifier resolves to an Ed25519 public key in the federation trust list (per §6.2).
       - `federation_signature` (required, string) — an unpadded base64url Ed25519 signature (86 chars, per §7.1b) over the canonical JSON (per §4a) of the Agent Card body with this field omitted. The signing key is looked up via `federation_credential` against the federation trust list.
    3. Add normative text: "The Agent Card is NOT self-signed. The v0.5 circular-signature construction (where a card claimed to sign itself with its own key) is retracted: a card signed by its own advertised key provides no third-party verifiability. In v0.5.1, cards are signed by a federation-scoped credential whose public key is distributed via the federation trust list. Trust-list distribution is federation-specific and out of scope for this specification beyond the interface boundary."
    4. Verification procedure: "To verify an Agent Card: (a) parse the card; (b) extract `federation_credential` and `federation_signature`; (c) construct the card body with `federation_signature` omitted; (d) canonicalize per §4a; (e) look up the public key for `federation_credential` in the federation trust list; (f) run `verify_strict` per §7.1b; (g) on any failure, reject the card with error `unauthorized`."
    5. Update any v0.5 example JSON block for Agent Card that this section inherits: replace `"signature": "..."` with `"federation_credential": "<opaque-id>", "federation_signature": "<base64url-sig>"`.
    6. Note that `federation_credential` is a BREAKING CHANGE from v0.5 and MUST be flagged in the changelog.

    **Changelog append:**

    Replace the `## §6.3 Card versioning` stub:

    1. "Every Agent Card MUST declare two integer version fields: `card_version` (monotonic non-negative integer incremented on every card publication) and `min_compatible_version` (non-negative integer, MUST be ≤ `card_version`, declaring the oldest card version whose bound commitments this card continues to honor)."
    2. Rotation semantics: "When an agent rotates its card from `card_version = N` to `card_version = N+1`, in-flight commitments that were bound to card version N remain valid through resolution provided the new card's `min_compatible_version ≤ N`. An agent that wishes to repudiate all outstanding commitments during rotation MAY set `min_compatible_version = N+1` on the new card; receivers MUST then reject any in-flight operation bound to card version ≤ N with error `unauthorized` (cause: `card_rotation_repudiated`)."
    3. Fresh vs bound: "Fresh requests (requests initiated AFTER the rotation is visible to the initiator) MUST use the latest card version. Bound operations (requests initiated BEFORE the rotation) use the card version referenced in the original commit per §11.2a."
    4. Cross-reference: "See §11.2a for the capability-snapshot binding rule that ties `card_version` to commit-time capture."
    5. Reject condition: "An Agent Card with `min_compatible_version > card_version` is malformed and MUST be rejected at parse."

    **Changelog append:**
    `v0.5.1-Δ12 — §6.3 Card versioning — CONTEXT D-14 — Pin card_version (monotonic int) and min_compatible_version (int); in-flight commits bound to card N survive rotation iff new card's min_compatible_version ≤ N; cross-linked to §11.2a.`
