<objective>
Ship `famp-inbox` — a pure-library crate that persists `SignedEnvelope` records to a JSONL file with
fsync-before-return durability, and reads them back with tolerance for a truncated tail line from a
mid-write crash. No network, no daemon, no CLI wiring — this plan ends at `cargo test -p famp-inbox`
green.

Purpose: Plan 02-02's `famp listen` needs a single, well-tested append primitive it can drop into an
axum handler. Keeping the inbox isolated in its own crate means the daemon plan can focus on signal
handling and TLS wiring instead of storage semantics.

Output: A new `famp-inbox` workspace member with public API (`Inbox::open`, `Inbox::append`,
`Inbox::read_all`, `InboxError`), unit tests for roundtrip + concurrent append serialization, and two
integration tests locking the truncated-tail and mid-file-corruption branches of the read path.
</objective>

```rust
// crates/famp-inbox/src/error.rs
#[derive(Debug, thiserror::Error)]
pub enum InboxError {
    #[error("io error at {path}")]
    Io {
        path: std::path::PathBuf,
        #[source]
        source: std::io::Error,
    },
    #[error("inbox line {line_no} is not valid JSON")]
    CorruptLine {
        line_no: usize,
        #[source]
        source: serde_json::Error,
    },
    #[error("inbox line contains embedded newline")]
    EmbeddedNewline,
}
```

```rust
// crates/famp-inbox/src/append.rs
use std::path::{Path, PathBuf};
use std::sync::Arc;
use tokio::fs::File;
use tokio::sync::Mutex;

pub struct Inbox {
    path: PathBuf,
    file: Arc<Mutex<File>>,
}

impl Inbox {
    /// Open (creating if absent, 0600) a JSONL inbox at `path`.
    pub async fn open(path: impl AsRef<Path>) -> Result<Self, crate::InboxError>;

    /// Append one already-serialized envelope as a single JSONL line.
    ///
    /// Contract: on `Ok(())`, `write_all` + `sync_data` have BOTH completed.
    /// The caller is free to return HTTP 200 after `Ok` observation.
    pub async fn append(&self, envelope_bytes: &[u8]) -> Result<(), crate::InboxError>;

    pub fn path(&self) -> &Path;
}
```

```rust
// crates/famp-inbox/src/read.rs
use std::path::Path;

/// Read every complete JSONL line from `path` as raw bytes.
///
/// Tail tolerance: if the LAST line is truncated (missing final `\n`) or fails
/// JSON sanity, it is silently skipped. Any non-terminal bad line yields
/// `Err(InboxError::CorruptLine)` — mid-file corruption is a hard error.
pub fn read_all(path: impl AsRef<Path>)
    -> Result<Vec<serde_json::Value>, crate::InboxError>;
```

  <behavior>
    - `Inbox::open(path)` creates the file at 0600 mode if absent (unix OpenOptions + mode(0o600)),
      opens it with `OpenOptions::append(true).read(false).write(true)`, and wraps the `tokio::fs::File`
      in `Arc<Mutex<File>>`.
    - `Inbox::open` returns `InboxError::Io { path, source }` on any FS error, never panics.
    - `Inbox::append(&self, bytes)` locks the mutex, rejects `bytes` containing any `\n` byte with
      `InboxError::EmbeddedNewline` BEFORE touching the file (prevents splitting one logical envelope
      into two JSONL lines), writes `bytes` then a single `\n`, calls `file.sync_data().await`, and
      returns `Ok(())` only after sync_data completes. This is the durability receipt.
    - `Inbox::path()` returns the stored path.
    - `read_all(path)` opens the file with `std::fs::File` (sync is fine — this is called from shutdown
      or cold-start code paths, not the hot HTTP handler), iterates `BufRead::split(b'\n')`, and for
      each line attempts `serde_json::from_slice::<serde_json::Value>`. Tail-tolerance rule: track
      whether the file ended with a `\n`; if it did NOT, the final chunk is a partial line — attempt
      to parse it, and on failure skip it silently. Any non-terminal parse failure returns
      `Err(InboxError::CorruptLine { line_no, source })` immediately.
    - Unit tests inline in append.rs (3):
      1. `append_then_read_back_roundtrip` — open inbox, append two raw JSON objects, `read_all` returns 2 values in order.
      2. `embedded_newline_rejected` — append fails with `EmbeddedNewline` without touching the file (file size unchanged).
      3. `concurrent_appends_serialize` — spawn 16 tokio tasks each appending a distinct 1 KB object,
         await all, assert `read_all` returns exactly 16 values and every object decodes cleanly (no
         interleaving). This locks the Arc&lt;Mutex&gt; serialization contract.
    - Add `crates/famp-inbox` to the `[workspace.members]` array in the root `Cargo.toml`.
    - Crate `Cargo.toml` deps: `tokio = { workspace = true, features = ["fs", "sync", "io-util", "rt"] }`,

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | `inbox.jsonl` on disk | accept | Single-user trust domain per v0.8 scope; 0600 mode is the only FS-level guard. Cryptographic integrity is already enforced by Ed25519 signatures inside the stored envelope — any tamper is detectable on re-read when the envelope is re-verified by Phase 3 consumers. |
| T-02-02 | Information Disclosure | `inbox.jsonl` readable by other users | mitigate | `Inbox::open` creates the file with unix mode 0600 explicitly via `OpenOptions::mode`. Covered by the roundtrip test verifying mode via `std::fs::metadata` (add to Task 1 unit test). |
| T-02-03 | Denial of Service | Unbounded file growth | accept | No rotation in Phase 2 per CONTEXT §Inbox Storage. Personal-runtime scope; rotation deferred to Phase 3+ if needed. |
| T-02-04 | Repudiation | Lost append after reported 200 | mitigate | `sync_data` called before `append()` returns Ok; Plan 02-02's SIGKILL durability test (`listen_durability.rs`) locks the end-to-end guarantee. |
| T-02-05 | Tampering | Mid-file corruption silently ignored | mitigate | `read_all` returns `CorruptLine` hard error on non-terminal parse failure (tested in `mid_file_corruption_is_hard_error`); only the physically-final truncated line is tolerated. |
