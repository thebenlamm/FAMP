<objective>
Wire the `famp listen` subcommand: load Phase 1 identity, bind a TCP listener (failing fast on port
collision), build an axum router that reuses `famp-transport-http`'s signature-verification middleware
but owns its own handler that appends to a `famp-inbox::Inbox` before returning 200, and shut down
cleanly on SIGINT/SIGTERM.

Purpose: This is the v0.8 daemon surface. Every later phase (send, await, inbox, MCP) assumes a
running listener that writes durable JSONL. The plan's single job is to stand that listener up with
fsync-before-200 durability and typed error paths.

Output: A working `famp listen` that a developer can start in a shell and send envelopes to; an
integration-test-friendly surface (listen::run takes a pre-bound listener for ephemeral-port tests in
Plan 02-03); two new `CliError` variants; `bin/famp.rs` upgraded to a multi-threaded tokio runtime.
</objective>

      1. Extracts the raw request `Bytes` body (signature has already been verified by the

      2. Calls `inbox.append(&bytes).await`.
      3. On `Ok`, returns `StatusCode::OK` (200 per success criterion 1 — note the default

         says "appears as a JSONL line within the same HTTP response cycle" and success criterion 1
         says "accepts POST" — we return **200 OK** as the durability-confirmed status, which is
         stricter than the 202 "received" the upstream transport uses. Document this deviation
         inline.)
      4. On `Err(InboxError)`, returns `500 Internal Server Error` and logs to stderr via
         `eprintln!` (tracing deferred to Phase 4 per CONTEXT §Deferred).
      - The router wraps the route with the same tower layering as
        `famp_transport_http::server::build_router`: outer `RequestBodyLimitLayer(1 MiB)` →
        `map_request(|r| r.map(Body::new))` → inner `FampSigVerifyLayer::new(keyring)` → handler.
        This preserves TRANS-04/07/09 (body limit + sig verification before routing) without
        modifying `famp-transport-http` source. The handler here differs from the upstream one only
        in the append-then-200 step — the middleware stack is byte-identical.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Remote peer → daemon HTTPS endpoint | Untrusted HTTP request crosses this boundary; sig-verify middleware is the gate. |
| Daemon → local filesystem | Inbox appends cross this boundary — 0600 mode inherited from Plan 02-01. |
| Daemon → OS signal delivery | SIGINT/SIGTERM cross this boundary; async signal handling must not deadlock the runtime. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Spoofing | Inbound envelope claiming wrong sender | mitigate | `FampSigVerifyLayer` (reused from v0.7 unmodified) verifies the Ed25519 signature against the keyring-resolved verifying key BEFORE the handler runs. Handler never sees an unverified envelope. |
| T-02-11 | Tampering | Altered wire bytes in transit | mitigate | TLS (rustls 0.23) terminates the connection; inside TLS, the envelope signature is an independent integrity gate. Two-layer defense. |
| T-02-12 | Repudiation | Server "lost" an inbound message | mitigate | `inbox.append` calls `sync_data` before returning Ok; handler returns 200 only after Ok. Plan 02-03 locks this with a SIGKILL durability test. |
| T-02-13 | Information Disclosure | Daemon leaks Ed25519 seed in stderr | mitigate | `CliError` variants carry no key material (Phase 1 D-05 invariant preserved). `eprintln!` in handler only logs `InboxError::Display`, which does not render byte contents. |
| T-02-14 | Denial of Service | Large request body exhausts memory | mitigate | `RequestBodyLimitLayer(1 MiB)` is the outermost tower layer — inherited from v0.7 TRANS-07. Requests > 1 MiB get 413 before the handler runs. |
| T-02-15 | Elevation of Privilege | Second `famp listen` silently uses a different port | mitigate | `TcpListener::bind` with explicit `AddrInUse` mapping to `CliError::PortInUse`; no fallback loop. OS-level bind is the lock. |
| T-02-16 | Denial of Service | Stuck handler prevents shutdown | accept | Inbox appends are bounded (single local fsync, milliseconds). A stuck handler is a bug, not a feature — we accept the rare case per CONTEXT §Graceful Shutdown. |
