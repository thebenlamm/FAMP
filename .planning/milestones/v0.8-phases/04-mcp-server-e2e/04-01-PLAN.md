<objective>
Replace Phase 3's two documented shortcuts with the real mechanisms the rest of
Phase 4 depends on: a multi-entry keyring built from `peers.toml`, and a
commit-reply handshake that drives the local FSM through REQUESTED →
COMMITTED → COMPLETED via real envelope round-trips.

Purpose: Every other Phase 4 deliverable (MCP server, two-daemon E2E, manual
witnessed smoke) depends on two daemons being able to sign-verify each other's
envelopes AND on the FSM walking the full three-state path under a real
round-trip. Phase 3 explicitly deferred both to Phase 4 and marked them
TODO(phase4) inline. This plan ships those corrections.

Output:
- `listen` daemon builds its keyring from `peers.toml` + self (not the hardcoded
  single entry).
- Inbound `request` envelopes trigger an auto-commit reply from the receiver.
- `famp await` walks the originator's local task to COMMITTED on seeing the
  commit reply.
- `fsm_glue::advance_terminal` is rewritten to step the FSM from its ACTUAL
  current state. The `__with_state_for_testing` shortcut is deleted from `src/`.
- Conversation integration tests that used the shortcut (Phase 3's
  `conversation_full_lifecycle`) are updated to drive the natural path.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| peer → daemon HTTPS inbox | Untrusted signed envelopes cross here; verified by FampSigVerifyLayer against the multi-entry keyring built in Task 1. |
| daemon → peer HTTPS (auto-commit outbound) | Reply envelopes cross here; pinned via TOFU fingerprint from `peers.toml`. |
| on-disk peers.toml | Mutable by `famp peer add`; read at daemon startup and (per re-read policy in Task 2) on each auto-commit dispatch. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | multi-entry keyring | mitigate | Unknown principals rejected by FampSigVerifyLayer (Test C in Task 1 locks this). `pubkey_b64` must decode to exactly 32 bytes; `KeyringBuildFailed` is fatal at daemon startup — no permissive fallback. |
| T-04-02 | Spoofing | auto-commit reply target | mitigate | Auto-commit looks up `req.from()` in peers.toml; principals with no registered peer entry get logged-and-dropped, never replied to. Prevents an attacker from coaxing the daemon into POSTing to an arbitrary endpoint. |
| T-04-03 | Tampering | commit-reply round-trip | mitigate | Reply envelopes are signed via the existing `FampSigningKey::sign` path; request causality binds `Relation::Commits` → original request id, preventing reply replay against a different task. |
| T-04-04 | Repudiation | fsm_glue natural walk | mitigate | Deleting `__with_state_for_testing` means the local record can only advance on a real signed envelope round-trip — the on-disk state history becomes an audit trail of actual protocol events, not synthetic seeds. |
| T-04-05 | Information Disclosure | peers.toml re-read | accept | peers.toml is 0600-owner-only; re-reading it per auto-commit dispatch is a minor fs-cost, accepted for simplicity. No secrets leak. |
| T-04-06 | Denial of Service | auto-commit storm | accept | An adversary with a registered principal could spam requests to provoke replies. Out of scope for v0.8 (one-long-task shape on localhost). Federation Profile (v0.9+) adds rate-limiting. |
| T-04-07 | Elevation of Privilege | await-side FSM advance | mitigate | `advance_committed` only runs on envelopes whose `class == Commit` AND whose `task_id` matches a local record in REQUESTED; any mismatch is a no-op. No privilege to advance an arbitrary task. |
</threat_model>
