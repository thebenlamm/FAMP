Satisfies BROKER-01 (UDS listener), BROKER-03 (single-broker exclusion via `bind()`), BROKER-04 (5-min idle exit), CLI-09 (disk mailbox via `famp-inbox`), and CLI-11 (`sessions.jsonl` diagnostic-only). This plan also lands the proto.rs additive `bind_as: Option<String>` field on `BusMessage::Hello` and the broker-side proxy semantics (per D-10) — every later CLI plan (02-03 register, 02-04 send, 02-05 inbox, 02-06 await, 02-07 join/leave/sessions/whoami) connects with `Hello { bind_as: Some(resolved_identity) }` to ride on a long-running registered holder, instead of carrying per-message `as` fields. Existing per-message `BusMessage` variants stay shape-stable — D-10 explicitly REJECTS the `BusMessage::Send { send_as }` field that the original plan 02-04 was going to add.

Purpose: Without this plan, no CLI subcommand can talk to anything, AND the proxy-Hello protocol that all later one-shot CLI commands depend on is not yet defined. The broker's `Out`-vec ordering invariant (Phase-1 D-04) is the load-bearing crash-safety property — `AppendMailbox` BEFORE `Reply(SendOk)`, `AdvanceCursor` AFTER `Reply(RegisterOk)` — and is enforced here at the executor.

Output: proto.rs Hello.bind_as additive field; broker/handle.rs proxy validation + effective_identity helper; broker/state.rs ClientState.bind_as field; `cli/broker/{mod,accept,idle,mailbox_env,cursor_exec,sessions_log}.rs` plus a `BrokerArgs` clap struct wired into `cli/mod.rs::Commands::Broker`.
