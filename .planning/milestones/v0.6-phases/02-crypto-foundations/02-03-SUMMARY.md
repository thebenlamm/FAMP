## What Shipped

The single most important artifact in Phase 2 is now on disk:
`crates/famp-crypto/tests/vectors/famp-sig-v1/worked-example.json`. It holds
the domain-separation prefix, the RFC 8032 Â§7.1 Test 1 keypair, the unsigned
envelope, the RFC 8785 canonical JSON (324 bytes), the full signing input
(336 bytes = prefix || canonical), and the Ed25519 signature â€” all as literal
hex strings transcribed from `FAMP-v0.5.1-spec.md` Â§7.1c.3 through Â§7.1c.6.
`PROVENANCE.md` documents the Python reference tooling (`jcs 0.2.1` +
`cryptography 46.0.7`) that originally produced those bytes and explicitly
forbids regenerating them from Rust (PITFALLS P10 â€” self-generated bytes
defeat the interop purpose of the fixture).

`tests/worked_example.rs` turns the fixture into a byte-exact gate.
`canonicalize_for_signature(parse(unsigned_envelope_json))` must equal
`hex::decode(signing_input_hex)` byte-for-byte, and
`verify_canonical_bytes(TrustedVerifyingKey::from_bytes(public_key_hex),
hex::decode(canonical_json_hex), FampSignature::from_bytes(signature_hex))`
must return `Ok(())`. A fixture-sanity guard also asserts
`signing_input_hex` starts with `domain_prefix_hex` and has the exact
`prefix_len + canonical_len` length, so any transcription error in the
fixture file fails fast with a clear message rather than a downstream
signature mismatch.

A dedicated `just test-crypto` recipe runs `cargo nextest run -p famp-crypto`
plus `cargo test -p famp-crypto --doc` and is now part of `just ci`. A
matching `test-crypto` job in `.github/workflows/ci.yml` runs the same
recipe and blocks the workspace test matrix on failure via
`needs: [test-canonical, test-crypto]`. Like Phase 1's canonical-JSON gate,
there is no `continue-on-error` â€” if the Â§7.1c byte-exact assertion fails,
nothing downstream ships.

The `famp-crypto` README is written to be the single source of truth for
the Phase 2 surface: it documents `DOMAIN_PREFIX` and the sanctioned
