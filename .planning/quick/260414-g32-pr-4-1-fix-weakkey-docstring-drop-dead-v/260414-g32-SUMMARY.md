# Quick Task 260414-g32: PR #4.1 — Adversarial Review Followups

Three small fixes surfaced by the adversarial review on PR #3 + PR #4
(zed-velocity-engineer, 2026-04-14), bundled into a single atomic commit
because all three belong to the same review session and none touches
load-bearing spec bytes.

## Fixes

1. **`CryptoError::WeakKey` rustdoc rewrite** — the prior wording
   ("Constructing a `TrustedVerifyingKey` that reached this variant
   means the trust boundary held") read as if construction succeeded.
   Reworded to make the semantics explicit: observing this variant in
   an error result means **no** `TrustedVerifyingKey` was constructed,
   so downstream `verify_strict` can assume a prime-order key. Spec §7.1b.

2. **Delete `CryptoError::InvalidSigningInput`** — grep-verified zero
   consumers across `crates/`. The "reserved for future use" comment
   contradicted the no-hypothetical-forward-compat rule. One dead arm
   removed from the public enum surface; no downstream match sites
   needed updating.

3. **CONTRIBUTING.md "Do Not Touch" bullet** — added
   `TrustedVerifyingKey::from_bytes` / `from_b64url` `is_weak()`
   ingress rejection to the load-bearing list. The reasoning captured
   in the bullet: `verify_strict` alone does **not** close the
   8-torsion subgroup hole; the ingress-time weak-key check in
   `TrustedVerifyingKey` construction is the actual gate. Spec §7.1b.
