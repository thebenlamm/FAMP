## Task 2 — `famp_inbox` no longer advertises an unimplemented `ack` (`4622ecb`)

Decision (user, 2026-07-08): **delete the false advertisement; do not wire
`ack` through.** Wiring it would make the primary MCP surface a writer of the

`crates/famp/src/cli/config.rs` (lines 23, 51, 72) — that guards `config.toml`

## Task 3 — WARN (not cap) on an oversized register/join drain (`6c1c971`)

### Threshold chosen

```rust
const DRAIN_WARN_BYTES: u64 = 8 * 1024 * 1024;   // 8 MiB — half of MAX_FRAME_BYTES
```

Exactly half of `famp_bus::codec::MAX_FRAME_BYTES` (16 MiB). This buys the
operator one full doubling of headroom before `RegisterOk.drained` /
`JoinOk.drained` overflows its single reply frame and registration itself
starts failing. The relationship is **pinned by a unit test** so that if the
frame limit ever moves, the warning headroom moves with it:

```rust
#[test]
fn drain_warn_threshold_is_half_the_reply_frame_limit() {
    assert_eq!(DRAIN_WARN_BYTES * 2, MAX_FRAME_BYTES as u64, …);
}
```

The warn fires **strictly above** the threshold (`> DRAIN_WARN_BYTES`), not at
it — also pinned by a test.

### `cap: None` IS PRESERVED. THE DRAIN IS NOT TRUNCATED.

This is stated explicitly, as the plan requires. `decode_lines` still passes:

```rust
&DrainPolicy {
    filter: &AwaitFilter::Any,
    skip_self_authored: None,   // preserved — self-addressed DMs must deliver
    cap: None,                  // preserved — NOT truncated
}
```

Capping would silently change Register/Join semantics: a client would come up
having never seen part of its own mailbox. That is a real design change and
belongs with backlog 999.11's retention work, not a quick win. This commit adds
an **operator-visible signal only**; every decodable record is still delivered.

The no-truncation contract is pinned by a dedicated test that feeds
`decode_lines` an oversized (>8 MiB span) two-record drain and asserts **two**
envelopes come out:

```rust
#[test]
fn oversized_drain_is_warned_but_never_truncated() { … assert_eq!(decoded.len(), 2, …) }
```

Also preserved: the missing-self-authored skip, and the existing per-record
WARN on undecodable lines (head-of-line resilience, fix 260611).

### Implementation

```rust
fn drained_span(start_offset: u64, records: &[DrainedRecord]) -> u64 {
    records.last().map_or(0, |last| last.end.saturating_sub(start_offset))
}
```

Computed purely from the offsets `DrainedRecord` already carries — **no framing
arithmetic is re-derived**, which is exactly what §3.2 removed. Empty slice
spans zero (guarded via `map_or`, not `unwrap`). `saturating_sub` prevents a
u64 wrap if `start_offset` ever exceeds the last record's `end` — an inverted
range reports a zero span, so the WARN cannot fire spuriously. **No
`unwrap`/`expect`** anywhere (repo denies both).

`warn_if_drain_oversized` emits exactly **one** `tracing::warn!` per drain,
carrying `mailbox`, `drained_bytes`, `records` count, and the real
`MAX_FRAME_BYTES` const as `limit`.

### Dependency + pinned-test constraints honored

- **`famp-bus` gained no new dependencies.** `just check-no-tokio-in-bus` →
  `OK - famp-bus is tokio-free.` The 6 new unit tests exercise `drained_span`
  as a pure function of `(start_offset, records)`, so no tracing-capture crate
  was needed (BUS-01).
- **Pinned property tests untouched and green:**
  ```

                            crates/famp-bus/tests/tdd02_drain_cursor_order.rs
  (empty)
  ```

Also updated the `decode_lines` doc comment: the old text said bounding the

## Task 4 — `BrokerState::view()` reuses `identity::canonical_holder_id` (`2b20568`)

### Which helpers were narrowed to `&BrokerState`

| Helper | Before | After | Why |
|--------|--------|-------|-----|
| `canonical_holder_id` | `&Broker<E>` | **`&BrokerState`** | Only reads `state.clients`; "liveness" here is the `connected` flag, not a process probe. |
| `registered_name` | `&Broker<E>` | **`&BrokerState`** | Only reads `state.clients`. (`#[allow(dead_code)]`, zero call sites — narrowed for consistency.) |

### Which genuinely need `&Broker<E>` because they touch `broker.env`

| Helper | Kept as | Reason |
|--------|---------|--------|
| `proxy_holder_alive` | `&Broker<E>` | Calls `broker.env.is_alive(pid)` — a real liveness probe through `BrokerEnv`. |
| `resolve_op_identity` | `&Broker<E>` | Delegates to `proxy_holder_alive` on the proxy branch, so it transitively needs `env`. |

A fourth helper, `effective_identity`, **already** took `&ClientState` and
needed no change. (The review's "three of four only need `&BrokerState`" count
is off by one in practice: two were narrowed, one was already narrow, two need
`env`.)

### Call-site migration — purely mechanical, no cascade

Six call sites, all a mechanical `broker` → `&broker.state`:

- `handle.rs:165` (`touch_activity`)
- `handle.rs:569` (`inbox`)
- `handle.rs:691` (`join`)
- `handle.rs:775` (`leave`)
- `handle.rs:825` (`whoami`)
- `awaiting.rs:104` (`resolve_await_owner`)

The plan's STOP condition ("if narrowing cascades beyond mechanical
`broker` → `&broker.state` at call sites, STOP and report") **was not
triggered.** No signature beyond the two helpers changed; no borrow-checker
restructuring was needed. `touch_activity` holds two simultaneous immutable
borrows of `broker.state` (one via `.clients.get()`, one via the narrowed
`canonical_holder_id(&broker.state, …)`) — legal, compiles clean.

### The deduplication

`state.rs` `view()` previously re-implemented the lookup inline:

```rust
    if s.connected && s.name.as_deref() == Some(bound) { Some(*id) } else { None }

```

Now:

```rust
crate::broker::identity::canonical_holder_id(self, bound).unwrap_or(*client_id)
```

Byte-for-byte the same predicate, same `unwrap_or` fallback — **zero behavior
change.** Verification:
