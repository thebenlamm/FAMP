## Task 1 — Align the test double with production (#12)

**files:** `crates/famp-bus/src/mailbox.rs`

**action:**
`InMemoryMailbox::drain_from` (`mailbox.rs:205-212`) returns
`Err(MailboxErr::CursorOutOfRange { requested, actual })` when `since_bytes > total`.
`DiskMailboxEnv::read_raw_from` (`crates/famp/src/cli/broker/mailbox_env.rs:182-187`)
instead returns `Ok(DrainResult { records: vec![], next_offset: file_len })` for
`since_bytes >= file_len`.

Make the double match production exactly:

- Replace the error branch with a clamp: when `since_bytes >= total`, return
  `Ok(DrainResult { records: Vec::new(), next_offset: total })`.
- Note the `>=` vs `>` change: production uses `>=`. At `since_bytes == total`
  both already produced an empty drain, but production reports
  `next_offset = file_len` while the double fell through to the loop and produced
  `next_offset = total` — same value. Confirm that before assuming it is a no-op.
- Delete `MailboxErr::CursorOutOfRange` entirely. Verified: zero references outside
  `mailbox.rs`. If the compiler disagrees, stop and report.
- Update `mailbox.rs:20-24`'s doc comment. Today it only claims the double mirrors
  `famp-inbox`'s *framing*. Extend it to state that the double mirrors production's
  **past-EOF semantics** too (clamp, never error), and say why: mailbox files
  legitimately shrink.

**Then add a conformance test** — the durable artifact of this task. A table-driven
test in `mailbox.rs` asserting `InMemoryMailbox::drain_from` returns
`(records: [], next_offset: total)` for `since ∈ {total, total + 1, total + 5000, u64::MAX}`.
Name it so its purpose is obvious, e.g. `past_eof_cursor_clamps_and_never_errors`.

## Task 3 — Clamp the cursor (#11)

**files:** `crates/famp-bus/src/broker/drain_walk.rs`, `awaiting.rs`, `handle.rs`

**action:**
`walk()` currently takes `(mailbox, since, records, policy)` and seeds
`let mut next_offset = since;` (`drain_walk.rs:86`). With zero records it returns
`since` — even when the drain reported the file now ends *before* `since`. `walk`
cannot see that, because it never receives `drained.next_offset`.

**Plumb it in.** Change `walk`'s signature to take the drain's end offset (e.g. a
`drain_next_offset: u64` parameter, or take `&DrainResult` directly — pick one and
be consistent across all four call sites). Seed:

```rust
// The mailbox can shrink beneath our cursor — `/famp-clear` truncates
// mailbox files while the broker holds in-memory offsets. The drain is
// authoritative about where the file now ends; a cursor past that point
// is stale, not an invariant violation. Clamp forward-progress-only.
let mut next_offset = since.min(drain_next_offset);
```

Verify this is a no-op on every non-truncation path before you trust it:
- records present → `drain_next_offset > since` → `min == since`. Unchanged.
- empty drain, file ends exactly at cursor → `drain_next_offset == since`. Unchanged.
- mid-line `since` → production snaps forward, `drain_next_offset == file_len >= since`
  → `min == since`. Unchanged (the loop then overwrites from `record.end`).
- **truncated** → `drain_next_offset < since` → `min == drain_next_offset`. Healed.

Then:
- `awaiting.rs:51` — `if batch.next_offset != since` now fires (`0 != 5000`) and
  writes the clamped cursor. Confirm; do not add a second guard.
- `handle.rs:601` — the channel loop's `if drained.records.is_empty() { continue; }`
  skips the write-back. Change it to write a clamped `inbox_offsets` entry when
  `drained.next_offset < cursor`, then continue.
- `awaiting.rs:256` — `debug_assert_eq!(next_offset, drained.next_offset)`. After the
  clamp an empty truncated drain satisfies it. **But reconsider the instrument:** a
  shrinking file is an expected external event, not a broker invariant. If it can
  still fire on any reachable state, replace it with a `tracing::warn!`. State your
  conclusion in the SUMMARY either way.
- Emit one `tracing::warn!` when a clamp actually happens (`drain_next_offset < since`),
  naming mailbox, stale cursor, and new cursor. Silent healing is how this stayed
  invisible.

**verify:**
