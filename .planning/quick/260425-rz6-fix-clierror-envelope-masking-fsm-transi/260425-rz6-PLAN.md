# Quick Task 260425-rz6: Fix `CliError::Envelope` Masking FSM Transition Errors

## Background

Adversarial review of the B2 fix saga (round 3) noted that when a task FSM
transition fails (e.g., `advance_committed` called on a task already in
COMMITTED), stderr surfaces a misleading top-line message: `"envelope
encode/sign failed"`. The actual reason — `IllegalTransition` from `famp-fsm` —
is shown only as a `caused by:` line via `std::error::Error::source` chaining
in `crates/famp/src/bin/famp.rs:44-49`.

Worse, the MCP error mapping at `crates/famp/src/cli/mcp/error_kind.rs:50`
returns `"envelope_error"` for FSM transition failures. MCP consumers may
switch on `famp_error_kind` to handle errors programmatically — they will
mis-handle FSM rejections as envelope problems.

Root cause: `crates/famp/src/cli/send/fsm_glue.rs` maps four distinct error
types into a single `CliError::Envelope` variant:

- Line 53: `parse_state` failure for an unknown task-state string → wrong
  variant; nothing envelope-related.
- Line 74: `advance_committed` FSM step failure → wrong variant.
- Line 96: `advance_terminal` FSM step failure → wrong variant.

The variant `CliError::Envelope` should mean what it says — envelope
encode/sign failed (the legitimate uses at `crates/famp/src/cli/send/mod.rs:415,418,481,482`).

## Truths

- `famp-fsm` is already a path dep of `famp` (`crates/famp/Cargo.toml:19`),
  so importing `famp_fsm::TaskFsmError` is free.
- `TaskFsmError` is `#[derive(thiserror::Error)]` with a single `IllegalTransition`
  variant whose Display is informative
  (`crates/famp-fsm/src/error.rs:11`).
- The `mcp_error_kind_exhaustive` test
  (`crates/famp/tests/mcp_error_kind_exhaustive.rs:159`) exercises the
  `Envelope` variant — it must continue to pass and gain coverage for the
  two new variants.
- Adding new `CliError` variants is **stable-API-compatible** — the
  `mcp_error_kind` match is already exhaustive (`#[must_use]` const fn,
  no `_ =>` arm), so the compiler will fail the build if a new variant is
  added without an arm. New `famp_error_kind` strings are additive: they
  do not break existing consumers, they just give them a new value to
  pattern on.
- No `Display` change to `Envelope` itself — the variant continues to mean
  exactly what its message says.

### T2: GREEN — add `FsmTransition` + `InvalidTaskState` variants and rewire fsm_glue

**Files:**
- `crates/famp/src/cli/error.rs`
- `crates/famp/src/cli/send/fsm_glue.rs`
- `crates/famp/src/cli/mcp/error_kind.rs`

**Action:**

1. Add to `CliError`:
   ```rust
   #[error("illegal task state transition")]
   FsmTransition(#[from] famp_fsm::TaskFsmError),

   #[error("invalid task state on disk: {value}")]
   InvalidTaskState { value: String },
   ```
   The `#[error]` line for `FsmTransition` is intentionally short — the
   inner `TaskFsmError`'s Display ("illegal transition: cannot apply
   class=... terminal_status=... from state=...") already carries the
   detail and is surfaced by main.rs's `caused by:` chain. The top-line's
   job is just to tell the operator *what category* of failure this is.
   Keeping the top-line short avoids duplicating the inner detail.

2. Add `mcp_error_kind` arms (alphabetical with the rest is fine — match the
   current ordering convention):
   ```rust
   FsmTransition(_) => "fsm_transition_illegal",
   InvalidTaskState { .. } => "invalid_task_state",
   ```
   And add to the `use` import block at the top of `error_kind.rs`.

3. In `crates/famp/src/cli/send/fsm_glue.rs`:
   - Replace the two `.map_err(|e| CliError::Envelope(Box::new(e)))?`
     calls (lines 74, 96) with `?` (the `#[from]` enables this).
   - Replace the `parse_state` error (line 53) with
     `CliError::InvalidTaskState { value: other.to_string() }`.
   - Update the rustdoc comments on lines 63 and 84 that say "mapped to
     `CliError::Envelope`" to say "surfaced as `CliError::FsmTransition`".

**Verify:**
- RED test from T1 now passes.
- `cargo test -p famp --test mcp_error_kind_exhaustive` passes (still needs a row
  for the new variants — see T3).
- `cargo build -p famp` succeeds.
- `cargo clippy --workspace --all-targets -- -D warnings` clean.

**Done when:** all three files updated, RED test green, committed atomically
with message `fix(quick-260425-rz6): add CliError::FsmTransition + InvalidTaskState variants; surface real reason`.
