Section title: `## How FAMP Signs a Message`

Content (one short paragraph each, in this order):

1. **Canonical JSON (RFC 8785).** One paragraph. FAMP payloads are signed over bytes, not over JSON values. Every signer and verifier must produce the same byte string for the same logical value ŌĆö that's RFC 8785 (JCS). Implemented in `famp-canonical`. Byte-exact is the entire interop story: if two implementations disagree on one byte of whitespace or key order, every downstream signature is unverifiable.

2. **Domain separation.** One paragraph. Before Ed25519 touches the canonical bytes, `famp-crypto` prepends a 12-byte constant: `b"FAMP-sig-v1\0"` (`DOMAIN_PREFIX`). Rationale: prevents a signature produced for a FAMP envelope from being replayed in any other context that also signs canonical JSON. Spec ┬¦7.1a, ┬¦╬ö01.

3. **The four steps.** Plain prose, not a code block. Sign = (1) canonicalize to JCS bytes, (2) prepend `FAMP-sig-v1\0`, (3) Ed25519 sign, (4) encode signature as base64url unpadded (`URL_SAFE_NO_PAD`, strict). Verify is the mirror: decode, canonicalize, prepend, Ed25519 **verify_strict** (never plain verify).

4. **INV-10.** One paragraph. Every envelope on the wire is signed. Unsigned envelopes are rejected at ingress. No exceptions, no "internal trusted" escape hatch. This is what makes non-repudiation an actual property of the system and not an aspiration.

5. **Task FSM.** One paragraph + an ASCII diagram. Five states, no intermediate parking. Use plain ASCII so it renders in any terminal:

```

       +--> COMPLETED

       |
       +--> CANCELLED
```

One sentence of context: each transition is a signed envelope; the FSM is enforced by `famp-fsm`; terminal states are `COMPLETED`, `FAILED`, `CANCELLED`.
