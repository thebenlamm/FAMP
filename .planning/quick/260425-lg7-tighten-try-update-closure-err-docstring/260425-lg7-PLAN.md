Round-3 adversarial review on `try_update` flagged a single MEDIUM finding: the
`# Guaranteed` section's closure-Err bullet claims "the on-disk file is
byte-identical to its pre-call state." That's a single-writer assertion the
function cannot deliver — the same docstring's `# NOT guaranteed` section
already disclaims locking, so a concurrent external writer can mutate the file
during the call. The byte-identity language leaks the single-writer mental
model into a guarantee `try_update` does not own.

Restate the closure-Err bullet in terms of what the function DOES (no call to
`write_atomic_file`), not what the file's state IS. Doc must be auditable
against the function body.

Purpose: protocol-grade honesty in primitive-crate rustdoc. Auditors should
never be able to point at a guarantee that the function cannot keep.

Output: one tightened bullet in `crates/famp-taskdir/src/store.rs`. Doc-only
change.
